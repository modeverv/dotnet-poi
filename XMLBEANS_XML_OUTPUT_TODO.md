# XMLBeans XML Output TODO

This file tracks the Phase -1 recovery work for XML lexical parity. The goal is to learn the XML output behavior that Apache POI gets through XMLBeans and encode only low-level lexical behavior in `PoiXmlWriter`.

## Decision

- Do not add Apache XMLBeans as a git submodule for now.
- Use the existing Apache POI submodule and Java interop Maven project to observe behavior.
- Prefer executable Java probes and fixture generation over reading XMLBeans internals first.
- If source-level XMLBeans inspection becomes necessary, use XMLBeans 5.3.0 source jars or the matching tagged source as a reference before considering a submodule.

POI currently declares XMLBeans 5.3.0 in `poi/build.gradle`, while the interop Maven project currently uses released POI artifacts through `tests/DotnetPoi.Interop.Tests/java/pom.xml`. Any probe result must record the POI/XMLBeans version it came from.

## Boundaries

Belongs in `PoiXmlWriter` or a narrow XML helper:

- XML declaration format and `standalone` behavior
- empty element form: `<tag/>` vs `<tag />` vs `<tag></tag>`
- attribute escaping and text escaping
- deterministic attribute order when callers provide attributes in POI/XMLBeans order
- namespace declaration spelling, placement, and duplication avoidance
- newline and whitespace emitted by XMLBeans/POI for saved OOXML parts

Does not belong in `PoiXmlWriter`:

- workbook-specific defined names
- Office revision GUIDs or user/local absolute paths
- fixture-specific relationship IDs or ordering unless proven to be POI model behavior
- unknown OOXML extension payloads
- semantic defaults that should be owned by XSSF/XWPF/XSLF model classes

Unknown or unsupported package parts should be preserved byte-for-byte where the public model does not yet represent them.

### Low-Level XML Writer Findings

These findings are candidates for focused `PoiXmlWriter` or narrow OOXML part writer tests:

- Spreadsheet XMLBeans parts such as `xl/workbook.xml`, worksheets, styles, shared strings, comments, and drawings write `<?xml version="1.0" encoding="UTF-8"?>` followed by a newline before the root element.
- OPC package-level XML parts such as `[Content_Types].xml`, `.rels`, and `docProps/core.xml` write `<?xml version="1.0" encoding="UTF-8" standalone="yes"?>` with no newline before the root element in the captured POI package output.
- Empty elements are consistently serialized without a space before the slash: examples include `<sheetData/>`, `<left/>`, `<b/>`, `<dxfs count="0"/>`, `<selection pane="bottomRight"/>`, `<a:fillRect/>`, and `<Relationship .../>`.
- XMLBeans/POI output is not indented. Apart from the declaration newline on spreadsheet XMLBeans parts and literal text content newlines, elements are written consecutively.
- Namespace declarations are written on the root for the namespaces needed throughout the part, using the default spreadsheet namespace for spreadsheet parts and explicit prefixes such as `r`, `xdr`, `a`, `cp`, `dc`, `dcterms`, and `xsi` where needed.
- Raw XMLBeans default save may use synthetic prefixes such as `main:` and local `rel:` declarations. POI/XMLBeans save options avoid that for spreadsheet parts by using the default spreadsheet namespace and root-level `xmlns:r` when relationship attributes are present.
- Attribute order follows POI/XMLBeans schema/caller order and must be preserved by the calling code. `PoiXmlWriter` should not sort attributes globally.
- Attribute escaping observed: `&` in relationship targets becomes `&amp;`; double quotes in format codes become `&quot;`; backslashes in format codes are preserved as literal backslashes.
- Text escaping observed: `&` becomes `&amp;`, `<` becomes `&lt;`, `>` remains literal `>` in text, and quotes/apostrophes remain literal in text.
- Whitespace-preserving shared string text uses `xml:space="preserve"` for leading spaces, trailing spaces, tabs, leading/trailing newlines, and whitespace-only text runs.
- Numeric lexical forms are caller/schema dependent. Captured output includes both `15` and `15.0`, `11` and `11.0`, and split values such as `2.0`; this should be treated as model/schema output, not a generic XML writer normalization rule.

### Not PoiXmlWriter Responsibilities

These findings are important, but should not become generic XML writer behavior:

- Workbook content such as defined names, print titles, `calcPr`, sheet order, `activeTab`, and `date1904` belongs to XSSF workbook/model code.
- Relationship IDs and relationship order are package/model behavior. Preserve caller order, but do not invent fixture-specific IDs in `PoiXmlWriter`.
- Styles, fonts, fills, borders, number formats, comments, drawings, hyperlinks, panes, merged regions, and page setup are semantic XSSF model responsibilities.
- Macro/VML/printer settings/media parts should be preserved byte-for-byte until their models are explicitly ported.
- Core properties timestamps and creator/modified metadata belong to OPC/core properties code, not the generic XML writer.

## Implementation Order

1. Add or update focused tests for XML declaration profiles:
   - XMLBeans spreadsheet part profile: UTF-8 declaration plus newline, no `standalone`.
   - OPC package part profile: UTF-8 declaration with `standalone="yes"` and no forced newline before the root.
2. Implement a small writer factory/profile layer around `PoiXmlWriter` rather than encoding one global declaration rule for all parts.
3. Add failing tests for empty element serialization without a space before the slash (`<tag/>`) for root, nested, prefixed, and attributed empty elements.
4. Implement the empty-element behavior in `PoiXmlWriter`. If `System.Xml.XmlWriter` cannot produce `<tag/>` directly, use a narrow stream/text interception layer and keep it covered by tests.
5. Add escaping tests for text and attributes:
   - text: `&`, `<`, `>`, quotes, apostrophes, tabs, and newlines
   - attributes: `&`, `"`, backslashes, relationship targets, and format codes
6. Implement only the escaping differences proven to diverge from `System.Xml.XmlWriter`; do not reimplement escaping if .NET already matches POI output.
7. Add namespace tests using default spreadsheet namespace plus `r`, and prefixed drawing namespaces such as `xdr` and `a`. Verify no duplicate namespace declarations and no synthetic `main:` prefix in POI-profile output.
8. Implement minimal namespace management in the writer/helper layer. Let callers declare namespaces in the POI/XMLBeans order; do not globally sort or hoist namespaces.
9. Add tests that verify attribute order is preserved exactly as callers write attributes. Do not add automatic attribute sorting to `PoiXmlWriter`.
10. Move to semantic XSSF tests using the generated POI integration fixtures: read values/styles/relationships/comments/drawings where supported, and preserve unknown parts where unsupported.
11. Only after semantic tests expose a concrete lexical mismatch, add another focused `PoiXmlWriter` test and implementation slice.
12. Keep higher-level writers model-driven and POI-source-backed; do not resurrect fixture-specific `XSSFWorkbook` XML payloads.

## POI Integration Fixture Plan

POI integration-level tests should improve fixture quality, but they should not be copied wholesale into dotnet-poi as byte-level parity requirements. The goal is to extract realistic scenario shapes from upstream POI and generate compact, deterministic fixtures.

Preferred source areas:

- `poi/poi-ooxml/src/test/java/org/apache/poi/xssf/usermodel/`
- `poi/poi-ooxml/src/test/java/org/apache/poi/xssf/model/`
- `poi/poi-ooxml/src/test/java/org/apache/poi/openxml4j/`
- `poi/test-data/`

Selection priority:

1. POI read/write round-trip tests.
2. Tests that verify package parts, relationships, or preservation behavior.
3. Historical bug tests that exercise realistic mixed OOXML structures.
4. Styles/shared strings/drawings/comments tests that combine multiple OOXML parts.
5. Macro-enabled preservation scenarios.

Porting approach:

1. Identify 5-10 upstream POI tests whose scenario shape is valuable and stable.
2. Record each selected upstream test path, method name, and reason for selection.
3. Recreate the scenario in `tests/DotnetPoi.Interop.Tests/java` as a fixture generator, not as a direct copy of POI's test assertions.
4. Emit both the `.xlsx`/`.xlsm` package and selected extracted XML parts.
5. Add C# tests that verify semantic compatibility first: values, styles, relationships, and round-trip readability.
6. Only add byte-level XML assertions when the difference is a proven `PoiXmlWriter` lexical behavior.

Do not generalize fixture-specific workbook content into `XSSFWorkbook`. If an upstream POI fixture contains unknown extension parts or unsupported model data, preserve those package parts byte-for-byte until the corresponding model behavior is ported.
