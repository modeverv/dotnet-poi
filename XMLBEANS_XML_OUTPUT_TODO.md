# XMLBeans XML Output TODO

This file tracks the Phase -1 recovery work for XML lexical parity. The goal is to learn the XML output behavior that Apache POI gets through XMLBeans and encode only low-level lexical behavior in `PoiXmlWriter`.

## Decision

- Do not add Apache XMLBeans as a git submodule for now.
- Use the existing Apache POI submodule and Java interop Maven project to observe behavior.
- Prefer executable Java probes and fixture generation over reading XMLBeans internals first.
- If source-level XMLBeans inspection becomes necessary, use XMLBeans 5.3.0 source jars or the matching tagged source as a reference before considering a submodule.

POI currently declares XMLBeans 5.3.0 in `poi/build.gradle`, while the interop Maven project currently uses released POI artifacts through `tests/DotnetPoi.Interop.Tests/java/pom.xml`. Any probe result must record the POI/XMLBeans version it came from.

## Boundaries

Belongs in `PoiXmlWriter` or a narrow XML helper:

- XML declaration format and `standalone` behavior
- empty element form: `<tag/>` vs `<tag />` vs `<tag></tag>`
- attribute escaping and text escaping
- deterministic attribute order when callers provide attributes in POI/XMLBeans order
- namespace declaration spelling, placement, and duplication avoidance
- newline and whitespace emitted by XMLBeans/POI for saved OOXML parts

Does not belong in `PoiXmlWriter`:

- workbook-specific defined names
- Office revision GUIDs or user/local absolute paths
- fixture-specific relationship IDs or ordering unless proven to be POI model behavior
- unknown OOXML extension payloads
- semantic defaults that should be owned by XSSF/XWPF/XSLF model classes

Unknown or unsupported package parts should be preserved byte-for-byte where the public model does not yet represent them.

## Initial Probe Observations

Generated by:

```bash
mvn test -f tests/DotnetPoi.Interop.Tests/java/pom.xml -Dtest=XmlBeansOutputProbeTest
```

Current output directory:

- `tests/DotnetPoi.Interop.Tests/fixtures/xmlbeans-output/`

Early observations from the first probe run:

- `XmlObject.save(OutputStream, ...)` writes `<?xml version="1.0" encoding="UTF-8"?>`.
- `standalone` is not emitted by these raw XMLBeans samples.
- XMLBeans writes a newline after the XML declaration.
- Empty elements are emitted as `<tag/>`, with no space before the slash.
- Plain default XMLBeans save uses a synthetic `main:` prefix for spreadsheet main namespace elements.
- `POIXMLTypeLoader.DEFAULT_XML_OPTIONS` changes the same samples to use the default namespace instead of `main:` prefixes.
- Attribute order follows the generated object/schema save order in the tested cases, e.g. page margins appear as `bottom`, `footer`, `header`, `left`, `right`, `top` after setting those values.
- Text escaping observed so far: `&` becomes `&amp;`, `<` becomes `&lt;`, while `>` in text remains `>` in the tested sample.

## POI Integration Fixture Plan

POI integration-level tests should improve fixture quality, but they should not be copied wholesale into dotnet-poi as byte-level parity requirements. The goal is to extract realistic scenario shapes from upstream POI and generate compact, deterministic fixtures.

Preferred source areas:

- `poi/poi-ooxml/src/test/java/org/apache/poi/xssf/usermodel/`
- `poi/poi-ooxml/src/test/java/org/apache/poi/xssf/model/`
- `poi/poi-ooxml/src/test/java/org/apache/poi/openxml4j/`
- `poi/test-data/`

Selection priority:

1. POI read/write round-trip tests.
2. Tests that verify package parts, relationships, or preservation behavior.
3. Historical bug tests that exercise realistic mixed OOXML structures.
4. Styles/shared strings/drawings/comments tests that combine multiple OOXML parts.
5. Macro-enabled preservation scenarios.

Porting approach:

1. Identify 5-10 upstream POI tests whose scenario shape is valuable and stable.
2. Record each selected upstream test path, method name, and reason for selection.
3. Recreate the scenario in `tests/DotnetPoi.Interop.Tests/java` as a fixture generator, not as a direct copy of POI's test assertions.
4. Emit both the `.xlsx`/`.xlsm` package and selected extracted XML parts.
5. Add C# tests that verify semantic compatibility first: values, styles, relationships, and round-trip readability.
6. Only add byte-level XML assertions when the difference is a proven `PoiXmlWriter` lexical behavior.

Do not generalize fixture-specific workbook content into `XSSFWorkbook`. If an upstream POI fixture contains unknown extension parts or unsupported model data, preserve those package parts byte-for-byte until the corresponding model behavior is ported.
