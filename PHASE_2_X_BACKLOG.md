# Phase 2.x Backlog

Phase 2 is intentionally a practical subset, not full Apache POI XSSF parity. Keep moving forward with small POI-compatible slices, and use this backlog for follow-up work before or alongside Phase 3.

Always compare with the original Apache POI source and add xUnit plus interop coverage for each item.

## Current Phase 2 Baseline

- Basic `XSSFCellStyle`
- Basic `XSSFFont`
- Built-in and custom number formats
- Indexed font colors
- Foreground fill color and fill pattern
- Basic border styles
- Cell `s` style index write/read
- POI Java can read the generated styled workbook fixture

## Important TODO - Style Byte-Level Parity

Phase 2 style support is still a subset. Because many style features remain unimplemented, full byte-level parity for style-related XML cannot be claimed yet.

Current guarantee:

- The XML writer parity layer is still in use.
- The implemented Phase 2 styled workbook is readable by Apache POI.
- Existing XML parity tests still pass.

Not yet guaranteed:

- `xl/styles.xml` generated by dotnet-poi is byte-for-byte identical to Apache POI/XMLBeans for the same style operations.
- Style-related worksheet XML is byte-for-byte identical for every style feature.
- Differences in style XML can be cleanly classified as writer divergence instead of missing style features.

TODO before claiming full style byte parity:

- Add Java fixture generators for each implemented style slice, such as bold font, custom number format, solid fill, and thin border.
- Save the POI-generated `xl/styles.xml` and relevant worksheet XML as fixtures.
- Generate equivalent dotnet-poi workbooks and compare the XML byte-for-byte for the implemented subset.
- Expand those fixtures as Phase 2.x fills in alignment, wrap text, border colors, fill background, RGB/theme colors, and style read parity.
- Only mark style byte parity complete after every supported style property has both semantic interop and byte-level fixture coverage.

## Phase 2.1 - Cell Types and Layout Basics

- Boolean cell public API and XML write/read
- Blank cell behavior aligned with POI
- Date cell API and serial date handling
- Error cell representation
- Row height

Suggested completion signal:

- POI writes boolean/date/blank/error examples and dotnet-poi reads them.
- dotnet-poi writes the same examples and POI reads them.
- Layout metadata survives a write/read round trip where applicable.

## Formula Boundary

Formula support should be split:

- Formula write/read: allow storing formula cells such as `A1+B1` and preserving cached values when present.
- Formula evaluation: defer to Phase 5.

Do not block Phase 3 on formula evaluation. It is acceptable to add formula write/read before Phase 5 if it helps compatibility, but record evaluator gaps explicitly.

## How To Pick Work

Prefer items that affect public API shape before Phase 3:

1. Cell types
2. Date handling
3. Merged regions and dimensions
4. Style properties that affect `ICell` / `IRow` / `ISheet`
5. Nice-to-have spreadsheet features

Avoid trying to exhaustively port all of XSSF before Phase 3. Apache POI is very broad; this project should advance by tested compatibility slices.
