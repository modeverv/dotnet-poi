- Not affiliated with the Apache Software Foundation

## Porting Phases

**Ship something working as early as possible.** Formula engine, styles, and charts come later.

### NuGet Package Strategy

The project is split into **two NuGet packages** to decouple Core stability from Formula maturity:

| Package | Contents | Stability target |
|---|---|---|
| **DotnetPoi.Core** | All format implementations (SS, POIFS, XSSF, HSSF, XWPF, HWPF, XSLF, HSLF) | **Stable v1.0** (release independently) |
| **DotnetPoi.Formula** | Formula evaluator (`FormulaEvaluator`) | **v0.x** (iterate freely) |

**Rules:**
- `Core` MUST NOT reference `Formula` in any way. Zero compile-time dependency.
- `Formula` references `Core` (it consumes Core types like `ICell`, `IWorkbook`).
- `Formula` registers itself with `Core` via `XSSFCreationHelper.RegisterFormulaEvaluatorFactory()` in its static constructor. At runtime, `createFormulaEvaluator()` uses lazy assembly discovery (`Type.GetType` + `RuntimeHelpers.RunClassConstructor`) to auto-detect `DotnetPoi.Formula`.
- All non-evaluator work (formula text read/write, cached value preservation, `setCellFormula`, `getCellFormula`, `calcPr fullCalcOnLoad`) lives in Core. Only actual evaluation lives in Formula.
- All new features go into Core first unless they are specifically formula evaluation logic.

**Implications for development:**
- Core can be stabilized and released as v1.0 without waiting for Formula.
- Formula tests (`DotnetPoi.Formula.Tests`) are independent of Core tests.
- `createFormulaEvaluator()` must always handle the missing-Formula case with `NotSupportedException`.
- When modifying Core interfaces that Formula depends on (e.g., `ICell`, `ICellStyle`), consider forward compatibility. Adding non-breaking members is safe; breaking changes require coordinated Formula updates.

Note: For now, formula evaluation and setting formula result values are intentionally omitted in the library; formulas may be preserved as text and cached results are only handled when present.

### Phase -1 (Foundation) — XML Output Parity

**This phase must be completed before Phase 0.** All subsequent phases depend on it.

#### Critical discovery: POI uses XMLBeans, not raw StAX

POI's OOXML XML output is not raw StAX. The actual stack is:

```
XSSF high-level API
    ↓
Apache XMLBeans (auto-generated classes from ECMA-376 XSDs)
    ↓
StAX (javax.xml.stream)
```

All `org.openxmlformats.schemas.*` classes are auto-generated by XMLBeans from the OOXML XSDs. This means **the ground truth for XML output is XMLBeans behavior, not raw StAX behavior**. XMLBeans has its own output characteristics — attribute ordering, namespace handling, empty element style — that sit above StAX and may differ from what StAX would produce natively.

This is a significant efficiency insight: rather than trying to match StAX behavior in the abstract, the fixture-based approach in this phase captures XMLBeans' actual output directly. `PoiXmlWriter` only needs to match what XMLBeans actually produces, which the fixtures define precisely. No need to read XMLBeans source or reverse-engineer its behavior — just run it and capture the output.

XML output behavior also differs subtly between XMLBeans/Java and .NET (`System.Xml.XmlWriter`). These differences are invisible at the logic level but corrupt interoperability at the byte level — the same class of problem as SAML signature verification failures caused by field ordering differences across implementations.

**Real-world evidence**: During implementation of AGILE encryption for OOXML, byte-level mismatch between Java and .NET XML output caused integrity verification to fail. The root cause was a single character difference in how empty elements are closed (`<tag/>` vs `<tag />`). The fix required hand-writing the XML as a string rather than relying on `XDocument.ToString()`. `PoiXmlWriter` in this project exists to solve this problem structurally so it never needs to be worked around again.

Known divergence areas to verify and resolve:

| Area | Java behavior | .NET risk | Fixable in wrapper? |
|---|---|---|---|
| Empty element closing | `<tag/>` (no space before slash) | `XDocument` may produce `<tag />` | ✅ Yes — write empty string before `WriteEndElement` |
| Empty element vs closing tag | `<tag/>` | `XmlWriter` may produce `<tag></tag>` or vice versa | ✅ Yes — override `WriteEndElement` |
| Zero/default values | Attributes with value `0` are written explicitly | May be omitted | ✅ Yes — always write explicitly |
| Attribute ordering | Consistent within POI | `XmlWriter` order may vary | ✅ Yes — write attributes in fixed order |
| Namespace declarations | Minimal, scoped | May be duplicated or hoisted | ⚠️ Partial — requires careful namespace management |
| Encoding declaration | `<?xml version="1.0" encoding="UTF-8"?>` | Encoding string may differ | ✅ Yes — `XmlWriterSettings` |
| Whitespace handling | Controlled explicitly | May insert/omit whitespace | ✅ Yes — `XmlWriterSettings.Indent = false` |

#### XmlWriterWrapper implementation notes

`XmlWriter` can be subclassed. The wrapper intercepts the following:

```csharp
public class PoiXmlWriter : XmlWriter
{
    private readonly XmlWriter _inner;

    // Key override: prevent space before slash in empty elements
    // XmlWriter has no built-in setting for this —
    // write an empty string before closing to force <tag></tag> form,
    // or suppress the space via stream interception if <tag/> form is needed.
    public override void WriteEndElement()
    {
        // Force Java-compatible closing behavior
        _inner.WriteFullEndElement(); // always writes </tag>, never <tag/>
        // OR intercept the raw stream to strip the space from <tag /> 
    }

    // Key override: always write zero-value attributes explicitly
    public override void WriteAttributeString(string localName, string value)
    {
        _inner.WriteAttributeString(localName, value); // never skip
    }
}
```

Note: if POI produces `<tag/>` (no space), stream-level interception of `XmlWriter` output may be needed as a last resort, since `XmlWriter` itself always writes `<tag />` with a space. Validate against fixtures first to confirm which form POI actually uses before implementing.

#### Approach

#### Why this is the foundation of everything

XML output parity is the lowest layer of the entire port. If this layer has divergence, every phase above it is affected — cell values, styles, images, encryption, formula output. Fixing it here means everything built on top works correctly by construction. When POI upgrades and changes its XML output, re-running the Java fixture generator automatically surfaces any new divergence, making this a living test suite rather than a one-time check.

#### Approach

**Step 1: Use LLM to generate a comprehensive POI fixture generator**

Ask the LLM to write a Java program using Apache POI that generates xlsx files covering all cases likely to produce XML output divergence between Java and .NET:

```
Write a Java program using Apache POI that generates xlsx files
covering the following cases exhaustively:

- Cells with value 0 (integer and float)
- Empty cells
- String, numeric, boolean, date, and formula cell types
- Multiple sheets
- Cases where multiple XML namespaces appear
- Deeply nested elements
- Empty elements
- Attributes with default values that POI writes explicitly
- Shared strings vs inline strings

The goal is to generate fixtures for detecting byte-level XML
output divergence between Java (StAX) and C# (XmlWriter).
Unzip each xlsx and save the internal XMLs as fixtures.
```

**Step 2: Run the generator in CI and capture fixtures**

The Java fixture generator runs as part of CI and saves raw XML files:

```
tests/DotnetPoi.Interop.Tests/
└── fixtures/
    └── xml-parity/
        ├── cell-zero-int.xml
        ├── cell-zero-float.xml
        ├── cell-empty.xml
        ├── cell-types.xml
        ├── multi-sheet.xml
        ├── namespaces.xml
        └── ...
```

**Step 3: Use LLM to identify divergence candidates**

Feed the captured XMLs to the LLM:

```
The following XML was produced by Apache POI (Java / StAX).
List every location where C# System.Xml.XmlWriter might produce
different byte-level output when reproducing this XML.

Focus on:
- Empty element closing style (<tag/> vs <tag />)
- Zero-value attributes that might be omitted
- Namespace declaration placement
- Encoding declaration differences

[paste XML here]
```

**Step 4: Implement PoiXmlWriter and write failing tests**

Each divergence candidate becomes a failing test. Fix `PoiXmlWriter` until all pass.

**Step 5: Gate Phase 0 on full parity**

`PoiXmlWriter` replaces all direct `XmlWriter` usage throughout the codebase — no exceptions.

**Do not start Phase 0 until all XML parity tests pass.**

### Phase 7 — Remaining Compatibility

Goal: close practical compatibility gaps left after the MVP. Work in this priority order:

1. **Round-trip correctness** — read a file, write it, read it back; assert every value (cells, styles, images) survives unchanged. This is the most fundamental correctness test.
2. **Bidirectional Java POI interop** — Java POI writes → dotnet-poi reads (correct parsing), dotnet-poi writes → Java POI reads (correct output). One test pair per supported format.
3. **API completeness** — fill in the POI API surface that each format needs.
4. **Macro-enabled file preservation** — when round-tripping xlsm/docm/pptm without touching macros, `vbaProject.bin` and all macro parts must survive byte-for-byte.
5. **XML byte-level parity** — only pursue this when a concrete interop failure cannot be fixed by semantic means. Do not chase attribute ordering or `standalone` differences as goals in themselves.

## Porting Procedure (Per Class)

Follow these steps in order. **Do not skip steps.**

### Step 1: Read the original

```
poi/src/java/org/apache/poi/{module}/{ClassName}.java
```

Understand the class's responsibility, dependencies, and key algorithms.

### Step 2: Read the corresponding Java test

```
poi/src/test/java/org/apache/poi/{module}/Test{ClassName}.java
```

If it exists, use it as the template for the ported test.

### Step 3: Generate the C# class

Conversion rules:

| Java | C# |
|---|---|
| `package org.apache.poi.X` | `namespace DotnetPoi.X` |
| `import` | `using` |
| `ArrayList<T>` | `List<T>` |
| `HashMap<K,V>` | `Dictionary<K,V>` |
| `Iterator<T>` | `IEnumerator<T>` |
| `instanceof` | `is` |
| `final` field | `readonly` |
| `@Override` | explicit `override` |
| `throws Exception` | XML doc comment only |
| Javadoc `/** */` | `/// <summary>` XML doc |
| `System.arraycopy` | `Array.Copy` |
| `byte[]` | `byte[]` (same) |
| `int` (unsigned use) | `uint` or annotated `int` |

### Step 4: Generate tests

- Framework: **xUnit**
- Port the Java test if one exists; otherwise write tests covering the main paths
- Test class name: `{ClassName}Tests`
- Test method name: `{Action}_{Condition}_{ExpectedResult}`

```csharp
[Fact]
public void Write_StringCell_ProducesValidXlsx()
```

### Step 5: Verify compatibility with Apache POI Java (Interop Tests)

The interop test suite lives in `tests/DotnetPoi.Interop.Tests/` and runs both directions in CI. **This is the most important test.**

```
Direction A: POI (Java) writes xlsx → dotnet-poi (C#) reads → assert contents match
Direction B: dotnet-poi (C#) writes xlsx → POI (Java) reads → assert contents match
```

The Java side is a self-contained Maven project under `tests/DotnetPoi.Interop.Tests/java/` with Apache POI as a dependency. Files exchanged between the two sides are placed in `fixtures/`.

## 移植フェーズ

**早期に動くものを出すことを最優先とする。** 数式エンジン・スタイル・グラフは後回し。

### NuGet パッケージ戦略

Core の安定度と Formula の成熟度を切り離すため、プロジェクトは **2つの NuGet パッケージ** に分割されています。

| パッケージ | 内容 | 安定度目標 |
|---|---|---|
| **DotnetPoi.Core** | 全フォーマット実装（SS, POIFS, XSSF, HSSF, XWPF, HWPF, XSLF, HSLF） | **Stable v1.0**（独立リリース可能） |
| **DotnetPoi.Formula** | 数式評価器（`FormulaEvaluator`） | **v0.x**（自由に反復） |

**ルール:**
- `Core` は `Formula` を一切参照してはいけない。コンパイル時の依存ゼロ。
- `Formula` は `Core` を参照する（`ICell`、`IWorkbook` などの Core 型を消費する）。
- `Formula` は静的コンストラクタで `XSSFCreationHelper.RegisterFormulaEvaluatorFactory()` を呼び出して Core に自己登録する。実行時に `createFormulaEvaluator()` は `Type.GetType` + `RuntimeHelpers.RunClassConstructor` による遅延アセンブリ探索で `DotnetPoi.Formula` を自動検出する。
- 非評価器の全作業（数式テキストの read/write、キャッシュ値保持、`setCellFormula`、`getCellFormula`、`calcPr fullCalcOnLoad`）は Core に置く。実際の評価ロジックだけが Formula に属する。
- 新機能は特に数式評価ロジックでない限り、まず Core に入れる。

**開発への影響:**
- Core は Formula を待たずに安定化・v1.0 リリースできる。
- Formula テスト（`DotnetPoi.Formula.Tests`）は Core テストから独立している。
- `createFormulaEvaluator()` は Formula がない場合に `NotSupportedException` をスローするケースを常にハンドルしなければならない。
- Formula が依存する Core インターフェース（`ICell`、`ICellStyle` など）を変更する際は前方互換性を考慮すること。非破壊的なメンバ追加は安全。破壊的変更は Formula 側の調整が必要。

注意: ライブラリ内での数式の評価と数式結果値の設定は当面オミットする。数式はテキストとして保持し、キャッシュ値がある場合のみ取り扱う。

### Phase -1（基盤）— XML 出力挙動の統一

**Phase 0 を始める前に必ず完了させること。** 以降の全フェーズがここに依存する。

#### 重要な発見: POI は生の StAX ではなく XMLBeans を使っている

POI の OOXML XML 出力は生の StAX ではない。実際のスタックはこうなっている:

```
XSSF 上位 API
    ↓
Apache XMLBeans（ECMA-376 XSD から自動生成されたクラス群）
    ↓
StAX（javax.xml.stream）
```

`org.openxmlformats.schemas.*` 以下のクラスは全て XMLBeans が OOXML の XSD から自動生成したものだ。つまり **XML 出力のグラウンドトゥルースは生の StAX の挙動ではなく XMLBeans の挙動**である。XMLBeans は StAX の上にあり、属性の順序・名前空間の扱い・空要素のスタイルについて独自の出力特性を持っている。

これは作業効率化の重要なポイントだ。StAX の挙動を抽象的に合わせようとするのではなく、このフェーズの fixture ベースのアプローチで XMLBeans の実際の出力を直接キャプチャすればよい。`PoiXmlWriter` が合わせるべき対象は XMLBeans が実際に出力するものであり、それは fixture が正確に定義してくれる。**XMLBeans のソースを読んだり挙動をリバースエンジニアリングする必要はない — 実行してキャプチャするだけでよい。**

さらに XMLBeans/Java と .NET（`System.Xml.XmlWriter`）の間にも細かい出力差異がある。ロジックレベルでは見えないが、バイト列レベルで相互運用を破壊する。SAML の署名検証がフィールド順の差異で通らなくなる問題と同じ構造。

**実戦での証拠**: OOXML の AGILE 暗号化実装時に、Java と .NET の XML 出力のバイト列不一致により整合性検証が失敗した。根本原因は空要素の閉じ方の1文字の差異（`<tag/>` vs `<tag />`）だった。最終的に `XDocument.ToString()` を信用せず XML を文字列として手書きすることで回避した。`PoiXmlWriter` はこの問題を構造的に解決し、二度と回避策が必要にならないようにするために存在する。

検証・解消すべき既知の差異:

| 項目 | Java の挙動 | .NET のリスク | wrapper で対応可能か |
|---|---|---|---|
| 空要素のスラッシュ前の空白 | `<tag/>` （空白なし） | `XDocument` は `<tag />` を出力する可能性がある | ✅ 可 — `WriteEndElement` の前に空文字を書く |
| 空要素 vs 閉じタグ | `<tag/>` | `XmlWriter` が `<tag></tag>` を出力する場合がある（またはその逆） | ✅ 可 — `WriteEndElement` をオーバーライド |
| ゼロ・デフォルト値 | 値が `0` でも属性を明示的に書く | 省略される可能性がある | ✅ 可 — 常に明示的に書く |
| 属性の順序 | POI 内で一貫している | `XmlWriter` の順序が変わる可能性がある | ✅ 可 — 固定順で書く |
| 名前空間宣言 | 最小限・スコープ内 | 重複または巻き上げられる可能性がある | ⚠️ 部分的 — 名前空間管理を慎重に行う |
| エンコーディング宣言 | `<?xml version="1.0" encoding="UTF-8"?>` | エンコーディング文字列が異なる可能性がある | ✅ 可 — `XmlWriterSettings` で制御 |
| 空白の扱い | 明示的に制御 | 自動挿入・省略の可能性がある | ✅ 可 — `XmlWriterSettings.Indent = false` |

#### PoiXmlWriter の実装方針

`XmlWriter` はサブクラス化できる。以下の箇所をオーバーライドする:

```csharp
public class PoiXmlWriter : XmlWriter
{
    private readonly XmlWriter _inner;

    // 重要: 空要素の空白を制御する
    // XmlWriterSettings に空白なし設定がないため、
    // WriteFullEndElement() で常に </tag> 形式にするか、
    // ストリームレベルで <tag /> の空白を除去する
    public override void WriteEndElement()
    {
        // Java と同じ閉じ方に統一
        _inner.WriteFullEndElement(); // 常に </tag> を出力、<tag/> にはならない
        // または: ストリームをインターセプトして <tag /> → <tag/> に変換
    }

    // 重要: ゼロ値の属性を常に明示的に書く
    public override void WriteAttributeString(string localName, string value)
    {
        _inner.WriteAttributeString(localName, value); // 省略しない
    }
}
```

注意: POI が `<tag/>`（空白なし）を出力する場合、`XmlWriter` 自体は常に `<tag />` を出力するため、ストリームレベルのインターセプトが最終手段として必要になる可能性がある。実装前に fixture で POI の実際の出力形式を確認すること。

#### なぜここが全ての基盤になるか

XML 出力のパリティは移植全体の最下層にあたる。この層に差異があると、その上の全フェーズ（セルの値・スタイル・画像・暗号化・数式出力）が影響を受ける。ここを固めることで、その上に何を積んでも正しく動くことが構造的に保証される。また POI がバージョンアップして XML 出力が変わった場合も、Java の fixture ジェネレーターを再実行するだけで新しい差異が自動的に検出される。**一度作ったら終わりではなく、生きたテストスイートとして機能し続ける。**

### Phase 8 — Interop 検証ゲート

目標: フォーマットの完了判定前に interop の必須確認項目を明文化する。

- **方向 A/B の両対応**: 各対応フォーマットで双方向テストを `tests/DotnetPoi.Interop.Tests/` に追加・維持する。
- **fixture 運用**: Java 出力 → `fixtures/from-poi/`、C# 出力 → `fixtures/from-dotnet-poi/` に固定し、ファイル名は決定的にする。
- **中核マトリクス**（該当する範囲で）: 値/型、数式（テキスト＋キャッシュ値）、スタイル、レイアウト（結合/幅/高さ/非表示/フリーズ）、共有文字列、ハイパーリンク、データ検証、条件付き書式、印刷設定、描画/画像。
- **マクロ形式**: xlsm/docm/pptm で `vbaProject.bin` と macro content type を byte-for-byte で保持する。
- **未知パーツ保持**: モデルが未対応の package part / relationship を round-trip で保全する。
- **XML パリティ**: 実害のある interop 失敗がセマンティックに直せない場合のみ byte-level fixture を追加する。

### Phase 9 — Documentation Site Generation

目標: Apache POI のドキュメントを参考資料として読みつつ、dotnet-poi 独自の現代的な Markdown ドキュメントを作り、検証済み examples を根拠に HTML 化して `docs/` に配置する。

このフェーズでは、実際に動く例・検証された出力・リポジトリが所有する Markdown を元に、利用者が信頼できるドキュメントサイトを作る。

ルール:

- **参照はするがコピーしない**: Apache POI の文書は Apache License 2.0 の著作物である。範囲・用語・利用者が知りたい内容を把握するために読むのはよいが、長文コピーや構造の丸写しは避ける。dotnet-poi の言葉で要約・再構成する。
- **`poi/` は編集しない**: Apache POI submodule は読み取り専用。参照元としてのみ使う。
- **docs は dotnet-poi 所有物にする**: Markdown とドキュメント生成用のソースファイルは `docs_src/` に置き、静的 HTML を `docs/` に生成する。
- **examples を先に作る**: ドキュメント化する前に `examples/` の実コードを作成・更新する。ドキュメントは compile/run 済みの例を参照する。
- **当面は使い方ドキュメントのみ**: 明示的に要求されるまで、Phase 9 では API リファレンス、クラス階層、アーキテクチャ解説ページを作らない。実行可能なサンプルに裏付けられた実用ガイドに集中する。
- **examples を検証する**: `dotnet`、Java、Python、その他必要なツールを使って、サンプルと生成物が正しく動くことを確認する。ドキュメント生成には、リポジトリから再現できる限り任意の言語・ツールを使ってよい。
- **API 情報は生成元を優先する**: public API を説明する場合、手書き一覧より C# ソースや生成メタデータから導いた情報を優先し、古くなりにくくする。
- **パッケージ分割を明確にする**: `DotnetPoi.Core` と `DotnetPoi.Formula` の違いを明記し、数式評価については意図的に限定された範囲以上の対応を示唆しない。
- **ソースと出力を分ける**: `docs_src/` を Markdown 原稿とドキュメント生成スクリプト/設定の標準配置場所にする。生成 HTML は `docs/` に置く。静的サイトジェネレーターがどうしても要求する場合を除き、Markdown 原稿と生成スクリプト/設定は `docs/` に置かない。runnable example code は引き続き `examples/` に置く。
- **スニペットはページ内に埋め込み、フルコードは examples へリンクする**: ドキュメントページには単機能を示す短いスニペット（概ね 20 行以内）を直接埋め込む。スニペットの直後またはページ末尾に `examples/` のフルコードへのリンクを置き、「全体を見たい場合はこちら」という形にする。長い複合サンプルは重複させず `examples/` にのみ置いてリンクする。

採用する実装方針:

- **言語/ツール**: 小さな C#/.NET のコンソール生成器を使い、独立した重いドキュメントフレームワークは使わない。既存ライブラリと同じ SDK で扱えるように、既存の `DotnetPOI.sln` に追加する。
- **生成器プロジェクト**: `tools/DotnetPoi.DocsGenerator/` に配置する。Markdown 変換には Markdig のような小さく一般的なパッケージは使ってよいが、必要になるまでは重い静的サイトスタックを避ける。
- **標準ソース構成**:
  - `docs_src/site.json` — サイト名、ナビゲーション順、バージョン/パッケージ注記、共通メタデータ。
  - `docs_src/content/` — 利用者向けセクション単位で整理した Markdown ページ。
  - `docs_src/content/getting-started/` — インストール、最初の workbook/document/presentation 例。
  - `docs_src/content/guides/` — xlsx、xls、docx、pptx、macro preservation、interop の形式別ガイド。
  - `docs_src/content/reference/` — 生成または C# ソース由来の API メモ。網羅的な API 一覧を手で保守しない。
  - `docs_src/content/compatibility/` — 互換性マトリクス、制限事項、Apache POI parity notes。
  - `docs_src/assets/` — 生成器が使う CSS、画像、小さな静的アセットのソース。
  - `docs_src/templates/` — 生成器が使う HTML レイアウトテンプレート。
- **生成出力構成**:
  - `docs/` — 生成された静的サイト専用。生成 HTML を手で編集しない。
  - `docs/assets/` — `docs_src/assets/` からコピー/最適化されたアセット。
- **ビルドコマンド**: `dotnet run --project tools/DotnetPoi.DocsGenerator -- docs_src docs`。
- **検証コマンド**: ドキュメント化した example project を先に実行し、その後 docs generator を実行する。完了扱いにする前に生成 HTML を spot-check する。
- **ソリューション方針**: docs generator はメインの `DotnetPOI.sln` に入れる。docs tool を独立リリースする必要が出るまでは別 solution を作らない。

推奨ワークフロー:

1. ドキュメントの章立てを作る: getting started、package installation、format-specific guides、examples、compatibility notes、limitations。
2. ドキュメント化するワークフローの runnable example を `examples/` に実装する。
3. examples と関連テストを実行し、説明する挙動が実際に成立することを確認する。
4. `docs_src/` に dotnet-poi 独自の言葉で Markdown を書く。単機能スニペットはページ内に直接埋め込み、スニペット直後またはページ末尾に `examples/` のフルコードへのリンクを置く。
5. 再現可能なスクリプトまたは静的サイトジェネレーターで HTML を `docs/` に生成する。
6. 生成された HTML を spot-check してから docs slice 完了とみなす。

## 1クラスの移植手順

各クラスは以下の手順で移植する。**必ずこの順序を守ること。**

### Step 1: 原典を読む

```
poi/src/java/org/apache/poi/{module}/{ClassName}.java
```

を読み、クラスの責務・依存関係・重要なアルゴリズムを把握する。

### Step 2: 対応するJavaテストを読む

```
poi/src/test/java/org/apache/poi/{module}/Test{ClassName}.java
```

が存在すれば読む。テストケースは移植テストの雛形にする。

### Step 3: C# クラスを生成する

変換ルール：

| Java | C# |
|---|---|
| `package org.apache.poi.X` | `namespace DotnetPoi.X` |
| `import` | `using` |
| `ArrayList<T>` | `List<T>` |
| `HashMap<K,V>` | `Dictionary<K,V>` |
| `Iterator<T>` | `IEnumerator<T>` |
| `instanceof` | `is` |
| `final` フィールド | `readonly` |
| `@Override` | `override`（明示） |
| `throws Exception` | XML doc に記載のみ |
| Javadoc `/** */` | `/// <summary>` XML doc |
| `System.arraycopy` | `Array.Copy` |
| `byte[]` | `byte[]`（同じ） |
| `int` (符号なし用途) | `uint` または注釈付き `int` |

### Step 4: テストコードを生成する

- フレームワーク: **xUnit**
- Java版テストが存在する場合はそれを移植する
- 存在しない場合は主要なパスをカバーするテストを新規作成する
- テストクラス名: `{ClassName}Tests`
- テストメソッド名: `{動作}_{条件}_{期待結果}` 形式

```csharp
[Fact]
public void Write_StringCell_ProducesValidXlsx()
```

### Step 5: Apache POI Java版との互換性検証（相互テスト）

相互テストスイートは `tests/DotnetPoi.Interop.Tests/` に置き、CI で両方向を自動実行する。これが**最も重要なテスト**。

```
方向A: POI（Java）で xlsx を書く → dotnet-poi（C#）で読む → 内容一致確認
方向B: dotnet-poi（C#）で xlsx を書く → POI（Java）で読む → 内容一致確認
```

Java側は `tests/DotnetPoi.Interop.Tests/java/` に置いた独立した Maven プロジェクトで、Apache POI を依存関係として宣言する。両側でやり取りするファイルは `fixtures/` に置く。

```
tests/DotnetPoi.Interop.Tests/
├── java/
│   ├── pom.xml                       # Apache POI 依存をここで宣言
│   └── src/test/java/
│       ├── WriteForDotnetTest.java   # 方向A: POI が fixture を生成
│       └── ReadFromDotnetTest.java   # 方向B: POI が C# 出力を検証
├── cs/
│   ├── ReadPoiGeneratedTests.cs      # 方向A: C# が POI fixture を読む
│   └── WriteForPoiTests.cs           # 方向B: C# が fixture を生成
└── fixtures/
    ├── from-poi/                # Java 生成ファイル（C# が読む）
    └── from-dotnet-poi/         # C# 生成ファイル（Java が読む）
```

CI で両方向を自動実行する:

```yaml
# .github/workflows/interop.yml
steps:
  - uses: actions/setup-java@v4
  - uses: actions/setup-dotnet@v4

  # 方向A: POI が書く → C# が読む
  - run: mvn test -f tests/DotnetPoi.Interop.Tests/java/pom.xml -Dtest=WriteForDotnetTest
  - run: dotnet test tests/DotnetPoi.Interop.Tests/cs/ --filter "Category=ReadFromPoi"

  # 方向B: C# が書く → POI が読む
  - run: dotnet test tests/DotnetPoi.Interop.Tests/cs/ --filter "Category=WriteForPoi"
  - run: mvn test -f tests/DotnetPoi.Interop.Tests/java/pom.xml -Dtest=ReadFromDotnetTest
```

## 命名規則

- **名前空間**: `DotnetPoi.{Module}` （例: `DotnetPoi.XSSF.UserModel`）
- **クラス名**: Java版と完全に一致させる（例: `XSSFWorkbook`）
- **メソッド名**: Java版に合わせる（ただし .NET 慣習で `GetXxx` はそのまま、プロパティ化は慎重に）
- **例外**: `IllegalArgumentException` → `ArgumentException` など標準.NET例外に対応させる

---

## 禁止事項

- `poi/` ディレクトリ内のファイルを編集しない（submodule は参照専用）
- Java版の動作と異なる独自実装を加えない
- 未実装のメソッドに黙って `throw new NotImplementedException()` を置かない → 必ずコメントで理由と追跡Issueを記載する
- テストなしでクラスをマージしない
